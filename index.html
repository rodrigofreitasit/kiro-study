00O que é o Kiro

Kiro é um IDE agêntico que traz estrutura ao desenvolvimento assistido por IA através de spec-driven development.

Kiro é um ambiente de desenvolvimento integrado (IDE) desenvolvido pela AWS, construído sobre a plataforma open-source Code OSS (a mesma base do VS Code). Lançado em preview público em julho de 2025, ele se posiciona como uma resposta estruturada às ferramentas de "vibe coding" como Cursor e GitHub Copilot.

A grande diferença do Kiro é que ele não é apenas um assistente de código. Ele funciona como um parceiro de desenvolvimento que transforma ideias em linguagem natural em especificações formais, documentos de design técnico e planos de implementação — antes mesmo de escrever a primeira linha de código.

Por que isso importa?

A maioria das ferramentas de IA para código gera soluções direto do prompt. Isso funciona para tarefas simples, mas gera problemas em códigos complexos: falta de contexto, decisões arquiteturais implícitas, dificuldade de rastrear o "porquê" das implementações. Kiro inverte essa lógica — a spec vira a fonte da verdade, e o código é um artefato gerado.

Importante

Apesar de ter sido lançado pela AWS, Kiro não é um serviço AWS e não exige conta AWS. Você pode fazer login com Google, GitHub, Builder ID ou AWS IAM Identity Center. Ele funciona de forma agnóstica com qualquer stack e qualquer provedor de nuvem.

03Vibe vs Spec

Vibe Coding — Exploração livre

Modo padrão, conversacional. Você descreve o que quer, o Kiro executa diretamente. Ótimo para tarefas rápidas, prototipagem e exploração de ideias sem pressão.

Use quando: a tarefa é simples, você está explorando uma ideia, prototipando ou não tem requisitos rígidos.

"Crie um endpoint Express que retorna uma lista de produtos mockados"

Resultado: código direto no editor, pronto para testar.

Spec Coding — Estruturado

Modo estruturado. Kiro gera primeiro os requisitos em notação EARS, depois o design técnico com diagramas, e então uma lista de tarefas rastreáveis. Só depois disso o código é escrito.

Use quando: a feature é complexa, envolve equipe, precisa de documentação, tem regressões custosas ou requer alinhamento entre produto e engenharia.

"Implemente um sistema de autenticação multi-tenant com JWT,
```

refresh tokens e rate limiting por usuário”

Resultado: 3 arquivos gerados (requirements.md, design.md, tasks.md) e execução passo-a-passo.

04Specs

Specs são o coração do Kiro. É onde a mágica do spec-driven development acontece.

A estrutura de uma Spec

Toda spec gera três arquivos que ficam versionados junto com seu código:

01
requirements.md

User stories e critérios de aceitação em notação EARS

→
02
design.md

Arquitetura, diagramas de sequência e decisões técnicas

→
03
tasks.md

Lista de tarefas discretas e rastreáveis para implementar

Notação EARS — Easy Approach to Requirements Syntax

O arquivo requirements.md usa EARS, um padrão que força clareza e testabilidade:

WHEN [condição ou evento]
```

THE SYSTEM SHALL [comportamento esperado]
```

Exemplos reais:

WHEN um usuário envia um formulário com dados inválidos
```

THE SYSTEM SHALL exibir erros de validação ao lado dos campos

WHEN o token JWT expirou
THE SYSTEM SHALL retornar 401 e limpar o refresh token do cliente

WHEN o usuário tenta fazer upload de arquivo maior que 10MB
THE SYSTEM SHALL rejeitar com erro 413 e mensagem “Arquivo muito grande”
```
Por que EARS funciona

Clareza: requisitos são inequívocos.
Testabilidade: cada requirement vira um caso de teste direto.
Rastreabilidade: você pode linkar cada linha de código a um requisito.
Completude: o formato força pensar em todas as condições.

Os dois tipos de Spec

Usadas para construir features novas. Kiro oferece dois workflows para Feature Specs:

Requirements-First: você começa descrevendo o comportamento desejado. Kiro gera requirements → design → tasks. Ideal para projetos greenfield ou product-led.

Design-First: você começa com arquitetura ou pseudocódigo. Kiro deriva os requisitos e tasks. Ideal quando há restrições técnicas rígidas (latência, compliance) ou quando você está portando designs de outras ferramentas.

Para diagnosticar e corrigir bugs com precisão, prevenindo regressões. Em vez de requirements.md, gera-se bugfix.md que documenta:

  • Comportamento atual (o bug)
  • Comportamento esperado
  • Comportamento que NÃO deve mudar (evita regressão)

Perfeito para bugs em sistemas críticos onde quebrar algo existente é caro.

Ciclo de execução

Com os três arquivos prontos, a aba de tarefas mostra cada item com status em tempo real. Você pode:

  • Executar tarefa por tarefa (clicando em "Start" em cada uma) — recomendado para revisar o que está sendo feito
  • Executar tudo de uma vez — mais rápido, mas com menos controle
  • Iterar na spec — editar requirements/design manualmente e pedir para o Kiro regerar as tasks

Boas práticas com Specs

  • Commite as specs no Git junto com o código. Elas são documentação viva.
  • Revise requirements antes de ir para design — é mais barato corrigir cedo.
  • Mantenha specs pequenas e focadas. Uma spec = uma feature. Se ficar gigante, quebre em várias.
  • Use Bugfix Specs para qualquer bug em produção — a documentação do "o que NÃO mudar" é ouro.
  • Não pule o design.md — mesmo em features pequenas, 5 minutos pensando em arquitetura economizam horas.

05Agent Hooks

Automações inteligentes que disparam baseadas em eventos. É como ter um colega júnior que nunca esquece de rodar os testes.

Como funcionam

Um Agent Hook é a combinação de dois elementos: um evento gatilho (trigger) e uma ação a ser executada. Quando o evento acontece, o Kiro executa automaticamente a ação em background — seja um prompt para o agente ou um comando shell.

Os 10 tipos de trigger disponíveis

TriggerQuando disparaUso típico
File SaveArquivo salvo que bate com um padrãoLint, format, rodar testes, gerar docs
File CreateArquivo novo criadoBoilerplate, licença, criar teste associado
File DeleteArquivo deletadoLimpar imports órfãos, remover testes
Prompt SubmitUsuário envia um promptAdicionar contexto, bloquear prompts, log
Agent StopAgente termina de responderCompilar código, rodar testes, revisar
Pre Tool UseAntes do agente usar uma ferramentaBloquear uso de certas tools, adicionar contexto
Post Tool UseDepois do agente usar uma ferramentaAuditoria, formatação pós-edição
Pre Task ExecutionAntes de iniciar tarefa de specSetup de ambiente, validar pré-requisitos
Post Task ExecutionApós completar tarefa de specRodar testes, atualizar docs, notificar
Manual TriggerExecução manual sob demandaCode review, security scan, otimização

Duas ações possíveis

// AÇÃO 01

Ask Kiro

Envia um prompt para o agente executar. Use para tarefas que exigem IA (gerar código, revisar, documentar).

// AÇÃO 02

Run Command

Executa um comando shell. Use para tarefas determinísticas (rodar lint, testes, scripts).

Duas formas de criar um hook

1. Descrever em linguagem natural — clique no "+" em Agent Hooks, escolha "Ask Kiro to create a hook" e descreva o que quer. Ex: "Toda vez que um arquivo .test.ts for salvo, rode o jest para esse arquivo e me avise dos erros".

2. Manualmente — preencha um formulário com título, descrição, evento, padrão de arquivo e a ação.

Exemplo real: auto-documentação

Título: "Atualizar docs ao salvar rota"
```

Evento: File Save
Padrão: “src/routes/**/*.ts”
Ação: Ask Kiro
Prompt:
“Analise a rota modificada e atualize o arquivo
docs/api.md com a nova especificação OpenAPI.
Preserve documentação de outras rotas.”
```

Exemplo real: qualidade automatizada

Título: "Rodar testes após tarefa de spec"
```

Evento: Post Task Execution
Ação: Run Command
Comando: “npm test – –changed”
```

Boas práticas com Hooks

  • Use padrões de arquivo específicos — **/*.ts é muito amplo. Prefira src/api/**/*.ts.
  • Cuidado com loops infinitos — um hook que salva arquivo em File Save pode disparar a si mesmo.
  • Prefira Run Command para tarefas determinísticas — mais rápido e previsível que usar o agente.
  • Hooks que consomem créditos devem ser pensados — cada prompt disparado conta no seu plano.
  • Compartilhe hooks via Git — eles ficam em .kiro/hooks/ no projeto.
  • Teste hooks manualmente antes — use Manual Trigger para validar antes de automatizar.
⚠ Atenção

Hooks em eventos frequentes (File Save em padrões amplos) podem consumir muitos créditos rapidamente e tornar o editor lento. Sempre comece restrito e expanda.

06Steering

Steering é a memória persistente do Kiro sobre o seu projeto. É onde você ensina "como as coisas funcionam por aqui".

O conceito

Steering files são arquivos Markdown que dão ao Kiro conhecimento contínuo sobre seu workspace. Em vez de repetir suas convenções em cada prompt ("use TypeScript estrito, prefira funções puras, nomes em camelCase..."), você escreve uma vez e o Kiro lembra para sempre.

Dois escopos: workspace vs global

ESCOPO LOCAL

Workspace

Fica em .kiro/steering/ na raiz do projeto. Aplica só àquele workspace. Ideal para convenções específicas do projeto.

ESCOPO GLOBAL

Global

Fica em ~/.kiro/steering/ (home do usuário). Aplica a todos os workspaces. Ideal para preferências pessoais gerais.

Em caso de conflito, o workspace vence — você pode sobrescrever regras globais por projeto.

Os 3 arquivos fundacionais

Kiro oferece gerar automaticamente três arquivos base, recomendados para todo projeto:

product.md
Produto

Propósito, usuários-alvo, features principais, objetivos de negócio

tech.md
Stack

Frameworks, bibliotecas, ferramentas e restrições técnicas

structure.md
Estrutura

Organização de arquivos, nomenclatura, padrões arquiteturais

Os 4 modos de inclusão

Nem todo contexto precisa estar sempre ativo. Kiro oferece 4 modos de inclusão configuráveis via front-matter YAML:

---
```

inclusion: always
—

Carregado em toda interação. Use para padrões universais do projeto: stack, segurança, convenções de código fundamentais.

—
inclusion: fileMatch
fileMatchPattern: [”**/*.ts”, “**/*.tsx”]
—

Só ativa quando o arquivo em contexto bate com o padrão. Use para guidance específico de camadas (API, componentes, testes).

—
inclusion: manual
—

Você invoca no chat com #nome-do-arquivo ou via /. Use para guias de troubleshooting, procedimentos de migração, referências pesadas.

—
inclusion: auto
name: api-design
description: REST API design patterns. Use when creating or modifying API endpoints.
—

Kiro lê a descrição e decide sozinho quando é relevante. Use para conhecimento especializado que sobrecarregaria o contexto se sempre ligado.

```

Exemplo prático de steering file

---
```

inclusion: fileMatch
fileMatchPattern: “src/api/**/*.ts”
—

# Padrões de API REST

## Estrutura de resposta
Toda resposta da API deve seguir o formato:

```json
{
  "data": {},
  "meta": { "page": 1, "total": 100 },
  "errors": []
}
```

## Códigos HTTP

- 200: Sucesso com conteúdo
- 201: Criação bem-sucedida
- 400: Erro de validação (incluir detalhes em errors[])
- 401: Não autenticado
- 403: Autenticado mas sem permissão

exceto `/auth/login` e `/auth/register`.
```

AGENTS.md — padrão aberto

Kiro suporta o padrão aberto AGENTS.md. Se você já tem um arquivo AGENTS.md na raiz do projeto (usado por outras ferramentas agênticas), o Kiro o reconhece automaticamente e sempre inclui no contexto.

Referências vivas a arquivos

Use a sintaxe #[[file:caminho/arquivo]] dentro do steering para linkar arquivos reais do projeto. O conteúdo é carregado dinamicamente:

Siga o padrão desta configuração:
```

#[[file:.env.example]]

Use como referência para novos componentes:
#[[file:src/components/ui/button.tsx]]
```

Boas práticas com Steering

  • Um domínio por arquivo: api-standards.md, testing-standards.md, security.md.
  • Nomes descritivos: api-rest-conventions.md > api.md.
  • Explique o porquê, não só o quê — contexto ajuda o agente a tomar decisões análogas.
  • Inclua exemplos de código — antes/depois, patterns e anti-patterns.
  • Nunca coloque segredos — steering vai no Git junto com o código.
  • Revise em mudanças arquiteturais — trate steering como código, exija PRs.

07MCP Servers

Model Context Protocol é o plugin system do mundo agêntico. É como dar superpoderes específicos para o Kiro.

O que é MCP

Model Context Protocol (MCP) é um protocolo aberto (criado pela Anthropic) que permite que agentes de IA se conectem a servidores externos especializados. Pense em MCP como extensões: cada servidor adiciona ferramentas, prompts e recursos específicos a um domínio.

Com MCP conectado, o Kiro pode:

  • Acessar documentação especializada — ex: servidor MCP da AWS dá acesso direto à documentação AWS
  • Consultar bancos de dados — MCP para PostgreSQL, MongoDB, etc.
  • Integrar com APIs externas — GitHub, Jira, Linear, Slack
  • Usar templates de prompt e recursos via menção # no chat
  • Executar ferramentas customizadas específicas do seu workflow

Exemplos de MCP Servers populares

AWS

AWS Docs MCP

Busca, leitura e recomendações direto da documentação AWS dentro do Kiro.

DB

Database MCP

Conecta a Postgres/MySQL/Mongo, executa queries e retorna schemas.

GIT

GitHub MCP

Cria PRs, lê issues, gerencia repositórios sem sair do editor.

API

Fetch MCP

Faz requisições HTTP para APIs externas e retorna dados estruturados.

Como configurar

  1. Abra as Settings com Cmd+, (ou Ctrl+,) e habilite MCP
  2. Crie um arquivo de configuração MCP (JSON) listando os servidores
  3. Vá à aba "MCP Servers" no painel Kiro para ver o status
  4. Clique em qualquer ferramenta listada para inserir um prompt template no chat

Exemplo de configuração MCP

{
```

“mcpServers”: {
“aws-docs”: {
“command”: “uvx”,
“args”: [“awslabs.aws-documentation-mcp-server@latest”]
},
“postgres”: {
“command”: “npx”,
“args”: [”-y”, “@modelcontextprotocol/server-postgres”],
“env”: {
“POSTGRES_URL”: “postgresql://user:pass@localhost/db”
}
}
}
}
```
⚠ Segurança

Servidores MCP podem acessar dados sensíveis. Trate-os como você trataria dependências: verifique a origem, revise o código antes de instalar, nunca use MCPs de fontes não confiáveis. Segredos e tokens devem usar variáveis de ambiente, nunca ficar hardcoded no arquivo de config.

Boas práticas com MCP

  • Comece com poucos servidores — cada MCP adiciona ferramentas que podem confundir o agente se forem muitas.
  • Verifique os logs em caso de erro — painel Output → "Kiro — MCP Logs".
  • Use MCPs oficiais quando disponíveis — AWS, GitHub, etc. mantêm servidores oficiais.
  • Documente no steering quais MCPs estão ativos e quando usá-los.

09Exemplos Práticos

Três exemplos de complexidade crescente para você sentir como o Kiro funciona na prática.

Exemplo 1 — Básico: Script de limpeza (Vibe Mode)

Cenário: você precisa de um script Python que remove arquivos temporários antigos de uma pasta.

# No chat, em Vibe Mode:
```

“Crie um script Python que remove arquivos .tmp e .log
com mais de 7 dias da pasta /tmp/app. Use pathlib e
mostre o que foi deletado. Inclua modo –dry-run.”
```

Kiro gera o script pronto, com argparse, logging e o --dry-run. Você revisa, aceita e usa. Tempo total: ~30 segundos.

Exemplo 2 — Intermediário: Endpoint com validação (Spec Mode)

Cenário: adicionar um endpoint de cadastro de usuário com validação completa numa API Express.

# Em Spec Mode, workflow Requirements-First:
```

“Implemente endpoint POST /users para cadastro com:
email único, senha com mínimo 8 caracteres e 1 número,
nome obrigatório (2-50 chars), idade opcional (18+).
Retornar erros de validação detalhados.
Usar PostgreSQL e bcrypt para senha.”
```

Kiro gera três arquivos. Veja o que sai em cada um (simplificado):

``` 
# Cadastro de Usuário

## User Story
Como visitante, quero me cadastrar com email e senha
para acessar funcionalidades autenticadas.

## Critérios de Aceitação (EARS)

WHEN o visitante envia POST /users com dados válidos
THE SYSTEM SHALL criar o usuário e retornar 201 com ID

WHEN o email já está cadastrado
THE SYSTEM SHALL retornar 409 Conflict com mensagem clara

WHEN a senha tem menos de 8 caracteres
THE SYSTEM SHALL retornar 400 com erro específico no campo

WHEN não há nome no payload
THE SYSTEM SHALL retornar 400 com "name is required"
```
``` 
# Design Técnico

## Arquitetura
Cliente → Express Route → Validator (zod) →
Controller → UserService → UserRepository → PostgreSQL

## Fluxo de criação
1. Router recebe POST /users
2. Middleware valida schema com Zod
3. Controller chama userService.create()
4. Service verifica email duplicado
5. Service hasheia senha com bcrypt (10 rounds)
6. Repository insere e retorna usuário sem senha

## Tratamento de erros
- Validação: 400 com array de errors por campo
- Duplicado: 409 com code USER_EMAIL_EXISTS
- Interno: 500 com logging (sem expor detalhes)
```
``` 
# Tarefas de Implementação

- [ ] 1. Criar schema Zod em src/schemas/user.ts
- [ ] 2. Criar UserRepository em src/repositories/user.ts
- [ ] 3. Criar UserService em src/services/user.ts
- [ ] 4. Criar UserController em src/controllers/user.ts
- [ ] 5. Registrar rota POST /users no router
- [ ] 6. Adicionar middleware de validação
- [ ] 7. Escrever testes unitários do service
- [ ] 8. Escrever testes de integração do endpoint
- [ ] 9. Atualizar documentação OpenAPI
```

Você clica em "Start" na tarefa 1, Kiro implementa, você revisa e aceita. Repete para cada tarefa. Ao final, todo o código está rastreado contra requirements específicos.

Exemplo 3 — Avançado: Combinando Specs + Hooks + Steering

Cenário: startup construindo um SaaS multi-tenant. Quer garantir qualidade, documentação sempre atualizada e padrões consistentes.

Setup inicial (uma vez só)

  1. Steering files:
    • product.md: descreve o SaaS, personas, modelo de negócio
    • tech.md: Node.js, Fastify, Prisma, PostgreSQL, Redis, TypeScript estrito
    • structure.md: modular monolith, DDD, camadas bem definidas
    • multi-tenancy.md (fileMatch em src/**/*.ts): regras de isolamento por tenant
    • testing.md (fileMatch em **/*.test.ts): padrões de teste com Vitest
  2. Agent Hooks:
    • File Save em src/**/*.ts → Run Command: npm run lint:fix
    • Post Task Execution → Ask Kiro: "Rode testes relacionados e reporte falhas"
    • File Create em src/features/**/*.ts → Ask Kiro: "Crie arquivo de teste correspondente"
  3. MCP Servers:
    • Postgres MCP — para queries diretas ao schema
    • GitHub MCP — para criar PRs direto do chat

Desenvolvimento diário

Para cada feature nova, o dev cria uma Feature Spec. Kiro já conhece o produto, stack, estrutura e regras de multi-tenancy por causa do steering. As tasks saem alinhadas ao padrão da casa. Hooks garantem lint, testes e documentação sempre em dia. Resultado: consistência automática em toda a equipe.

Observação

Este setup foi validado por times reais em produção, incluindo startups que relatam semana para dia em time-to-value com spec-driven development bem configurado.

10Boas Práticas

Para todos os projetos

  • Sempre crie os 3 steering fundacionais no início de um projeto — é o melhor investimento de 10 minutos que você fará.
  • Commite a pasta .kiro/ — specs, steering e hooks são parte do projeto.
  • Itere em specs antes de codar — 15 minutos ajustando requirements economizam 3 horas de código errado.
  • Use Bugfix Specs para qualquer correção em produção — a seção "comportamento que não deve mudar" é o que previne regressões.
  • Revise o que o agente faz — Kiro é bom, não é infalível. Sempre revise diffs antes de aceitar.

Para times

  • Padronize steering files entre devs — use o mesmo .kiro/steering/ para todos.
  • Defina donos de steering files — como se fossem módulos de código.
  • Revise specs em PRs — specs mudam? Isso é mudança de comportamento, merece review.
  • Use hooks para enforcement automático — mais barato que code review manual.
  • Considere o plano Enterprise em times com 10+ devs pela gestão centralizada.

Para gestão de créditos

  • Mantenha o modelo em Auto — o roteamento inteligente economiza muito.
  • Use Vibe Mode para tarefas simples — spec consome mais créditos que vale a pena só em tarefas complexas.
  • Evite hooks em File Save com padrões amplos — podem disparar centenas de vezes por dia.
  • Monitore o dashboard — créditos são atualizados a cada 5 minutos.

11O que Evitar

Armadilhas comuns

Spec gera 3 arquivos, design documentado, lista de tarefas. Isso é ótimo para features de verdade — mas é overkill para um script de 20 linhas. Use Vibe Mode para tarefas simples. Não tenha medo de "ser rápido" — nem tudo precisa virar processo formal.

Um único steering.md com 2000 linhas é pior que não ter steering. O agente perde foco e o contexto fica barulhento. Mantenha um arquivo por domínio, use fileMatch para reduzir ruído.

Por mais estruturado que seja o spec-driven, o código ainda é gerado por IA. Bugs sutis, dependências erradas, vulnerabilidades de segurança acontecem. Revise sempre os diffs, especialmente em código que vai para produção ou lida com segurança.

Um hook em File Save que modifica o mesmo arquivo salvo dispara a si mesmo infinitamente. Sempre pense no ciclo: o que minha ação modifica? Pode disparar outro hook? Teste com Manual Trigger antes de ativar.

Steering files vão no Git. Config do MCP pode ir no Git. Nunca coloque API keys, senhas ou tokens em texto plano. Use variáveis de ambiente, .env (gitignored) ou secret managers.

Tarefas de spec consomem mais que prompts simples. Hooks disparam frequentemente. Opus consome mais que Haiku. É fácil queimar um mês inteiro em uma semana. Acompanhe o dashboard semanalmente, especialmente no início.

Vibe coding um bug fix frequentemente introduz regressão. A documentação "comportamento que NÃO deve mudar" da Bugfix Spec é o ativo mais valioso para correções em código legado.

MCP servers têm acesso a muito — seu sistema de arquivos, sua rede, seus dados. Só instale MCPs de fontes conhecidas (oficiais ou auditáveis). Revise o código-fonte antes em casos sensíveis.

Sinais de que algo está errado

  • 🚩 Kiro gera código que ignora suas convenções → seu steering está incompleto ou mal escrito
  • 🚩 Specs ficam grandes demais → você está tentando fazer muita coisa em uma feature só
  • 🚩 Créditos sumindo rápido → revise hooks e modelos em uso
  • 🚩 Agente executa comandos inesperados → revise regras de Pre Tool Use
  • 🚩 Código repetitivo apesar do steering → reforce exemplos no steering, use inclusion: always em fundamentais

O que é um crédito?

Um crédito é uma unidade de trabalho em resposta a um prompt. É medido com duas casas decimais (mínimo 0,01).

  • Prompts simples: menos de 1 crédito cada
  • Execução de spec task: tipicamente mais de 1 crédito
  • Modelo Auto: custo base (1x)
  • Claude Sonnet 4: ~1,3x mais caro que Auto
  • Opus: ainda mais caro

Tudo consome créditos

Qualquer operação agêntica conta: vibe mode, spec mode, refinamento de specs, execução de tarefas, e hooks quando disparam. Por isso hooks em eventos frequentes podem ser caros.

Bônus e overage

Trial

Primeiro login = 500 créditos bônus, válidos por 30 dias. Aplica mesmo no plano Free.

Overage: em planos pagos, você pode ativar cobrança adicional a $0,04/crédito quando estourar. Vem desativado por padrão — ative em Settings se quiser continuidade sem limite.

Enterprise

Para times, o plano Enterprise oferece:

  • Billing centralizado
  • SSO via SAML/SCIM (AWS IAM Identity Center)
  • Dashboard de gestão organizacional
  • Analytics de uso
  • Controles de segurança e privacidade enterprise-grade
  • Atribuição de planos por usuário
Atenção aos detalhes

Créditos não acumulam — reiniciam todo mês.
GovCloud (US) custa ~20% a mais e não tem plano Free.
Subscriptions são individuais — não compartilhe sua conta com o time.

KIRO

14Limites & Considerações

Limites técnicos

  • Suporte a linguagens: Kiro suporta Python, Java, JavaScript, TypeScript, C#, Go, Rust, PHP, Ruby, Kotlin, C, C++, shell, SQL, Scala, JSON, YAML e HCL (entre outras). Linguagens menos comuns podem ter qualidade reduzida.
  • Context window: mesmo com Claude Sonnet 4.5, há limite de contexto. Projetos muito grandes podem exigir estratégia — use steering com fileMatch para reduzir ruído.
  • Qualidade = qualidade do input: prompts vagos = resultados vagos. Specs mal formuladas = código mal estruturado.
  • Créditos não rolam: não usou o mês? Perdeu.

Limitações de produto

  • Extensões: usa Open VSX (não o marketplace oficial da Microsoft). A maioria das extensões populares existe, mas algumas podem faltar.
  • Compartilhamento de conta: não permitido. Cada dev precisa da sua.
  • GovCloud: não tem Free tier e custa 20% a mais.
  • Regionalidade: modelos disponíveis variam por país (restrições dos provedores de LLM).

Limitações conceituais

  • Spec-driven tem custo cognitivo: pensar em requirements antes de codar é diferente. Alguns devs preferem a liberdade do vibe coding.
  • Não substitui conhecimento: você ainda precisa entender o que Kiro gera para revisar bem.
  • Curva de aprendizado real: usar Kiro bem (steering + hooks + specs em harmonia) leva semanas de prática.
  • Preview status: algumas features ainda são preview. Mudanças podem acontecer.
Privacidade

Kiro tem controles enterprise-grade de segurança e privacidade. Detalhes completos em kiro.dev/docs/privacy-and-security. Verifique com o time de segurança antes de usar em código proprietário sensível.

15Glossário

Agent Hook
Automação disparada por evento (salvar arquivo, criar, prompt submetido, etc.) que executa prompt ou comando shell.
Agentic IDE
IDE onde a IA não apenas autocompleta, mas age com autonomia — executa tarefas, usa ferramentas, toma decisões.
AGENTS.md
Padrão aberto para arquivo de instruções a agentes de IA, reconhecido por várias ferramentas incluindo Kiro.
Autopilot
Modo onde Kiro executa tarefas grandes autonomamente, sem precisar de instrução passo-a-passo.
Code OSS
Versão open-source do Visual Studio Code, sem telemetria e branding da Microsoft. Base do Kiro.
Crédito
Unidade de consumo no Kiro. Cada prompt consome uma fração, variando com complexidade e modelo.
EARS
Easy Approach to Requirements Syntax. Notação estruturada (WHEN/SHALL) usada pelo Kiro para requirements.
Inclusion Mode
Configuração de quando um steering file é carregado: always, fileMatch, manual ou auto.
MCP
Model Context Protocol — protocolo aberto para conectar agentes de IA a servidores externos com ferramentas e dados.
Spec
Especificação estruturada no Kiro, composta por requirements.md, design.md e tasks.md.
Spec-driven development
Metodologia onde a especificação é a fonte da verdade e o código é derivado dela.
Steering
Sistema de arquivos markdown que dão ao Kiro conhecimento persistente sobre projeto, convenções e padrões.
Vibe coding
Estilo de desenvolvimento onde você conversa livremente com a IA para gerar código sem estrutura formal.

16Trilha de Aprendizado

Sugestão de ordem para dominar o Kiro em 3 semanas.

Semana 1 — Fundação

  • Dia 1-2: Instale, explore a interface, faça o tutorial "Your first project" oficial.
  • Dia 3-4: Use apenas Vibe Mode. Tente tarefas que você faria normalmente. Sinta como responde.
  • Dia 5-7: Crie seu primeiro Spec (algo simples — um endpoint, um componente). Vá do requirements ao código. Observe os 3 arquivos gerados.

Semana 2 — Personalização

  • Dia 8-9: Gere os 3 steering fundacionais para um projeto real seu. Itere até ficarem bons.
  • Dia 10-11: Crie 2-3 steering files específicos (api-standards, testing, etc.) com fileMatch.
  • Dia 12-14: Experimente Agent Hooks. Comece com 1 hook simples (ex: formatar ao salvar). Depois adicione 2-3 mais úteis.

Semana 3 — Escala

  • Dia 15-16: Conecte um MCP server (sugestão: AWS Docs ou GitHub). Use-o em um fluxo real.
  • Dia 17-18: Use Bugfix Spec em um bug real. Observe a documentação "não deve mudar".
  • Dia 19-21: Combine tudo: projeto novo com steering completo + hooks + specs + MCP. Compare a produtividade com uma semana sem Kiro.

Além da trilha

  • Siga o blog oficial em kiro.dev/blog para ver patterns e casos reais
  • Leia o FAQ completo em kiro.dev/faq
  • Explore exemplos abertos no GitHub (procure "Kiro spec examples")
  • Experimente o Kiro CLI em workflows CI/CD
